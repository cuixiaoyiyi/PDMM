① Overview & headline results

PDMM_deliverable/ · see RESULTS.md for the full prose

This artifact implements and empirically evaluates the ICSE submission “Constructing Precise Dummy Main Methods for Android via Hierarchical Lifecycle Modeling”, built on top of DMMPP (Cui et al., ISSTA 2024) and the FlowDroid / Soot / Androlic stack. What was actually built
  • A measurement layer that re-uses DMMPP's unmodified front-end to score RQ1 + RQ2 on any APK.
  • A real de-bloated DMM generator that emits valid, re-loadable instrumented APKs — and fixes two crash bugs in the shipped DMMPP.
  • A wiring of that generator into DMMPP's real downstream analyzer (Androlic) to obtain the RQ3 “Ours” column on the same engine.

② Environment & reproduction

PDMM_deliverable/scripts/ · src/

No Maven needed. DMMPP ships a fat jar (DMMPP.jar, contains Soot + FlowDroid 2.10); Androlic ships Androlic_DMMPP.jar (adds the jymbolic SymbolicEngine). The new sources compile directly against those jars with JDK 8.

# compile (metrics / generator / FlowDroid RQ3)
javac -cp DMMPP.jar -d build  $(find src -name '*.java')

# RQ1 + RQ2 over all apps  ->  results/rq_app_summary.csv , rq1_components.csv
bash scripts/run_batch.sh ;  python scripts/aggregate.py

# RQ3 FlowDroid baseline    ->  results/rq3_flowdroid.csv , rq3_flowdroid_real.csv
bash scripts/run_rq3.sh ;  bash scripts/run_rq3_real.sh ;  python scripts/rq3_analyze.py

# Emit de-bloated APKs       ->  out_apk/PDMMPP_*.apk , results/rq1_generated.csv
bash scripts/run_gen.sh

# RQ3 "Ours" via Androlic    ->  results/rq3_androlic.csv   (compile against Androlic_DMMPP.jar)
bash scripts/run_androlic.sh ;  bash scripts/run_androlic_more.sh

CJK-path caveat: the Windows-built Androlic jar mis-decodes non-ASCII paths in its config reader, so the Androlic RQ3 step runs from an ASCII working dir (C:\pdmm_work\). All other steps run in place.

RQ1 · Parameter de-bloat

src/cn/ac/ios/pdmm/ParamPartitioner.java · results/rq_app_summary.csv

For each component we collect the exact multiset of formal parameters the baseline DMMPP allocates at the dummy-main entry (one per callback/lifecycle argument), then count how many symbolic variables survive equivalence partitioning. PRR = (|Pbase| − |Pours|)/|Pbase|.

Dataset#appsbaselineconservativeaggressivePRR (con)PRR (agg)
DroidBench188125439537968.5%69.8%
F-Droid10123746643362.3%65.0%
Google Play1067227625858.9%61.6%
ALL20831631137107064.1%66.2%

Finding. Aggressive partitioning removes 66.2% of formal parameters overall — matching the paper's claimed 64.5%. Most bloat is intra-lifecycle (a single saved-state Bundle repeated across onCreate/onStart/onRestoreInstanceState/onPostCreate/onSaveInstanceState), which is why even the 100%-safe conservative strategy already removes ~64%.

vs paper: our conservative (64.1%) is stronger than the paper's 28.4% because we merge all same-type backbone parameters (provably the same instance), not only directly dominating pairs. The aggressive figure matches closely.

The reduction is realised in actual emitted code too — see “Real de-bloated generator” (2865→974 = 66.0% over generated, JVM-verified bytecode).

RQ2 · Lifecycle Consistency Score (LCS)

src/cn/ac/ios/pdmm/LifecycleSpec.java · LcsEvaluator.java · results/rq_app_summary.csv

Why DMMPP = 1.000 on activity-only apps but 0.857 with fragments

DMMPP's lifecycle model is app-independent. Decomposed:

  • Activity graph: 13 edges, all valid → 1.000.
  • Fragment graph: 15 edges, 4 invalid → 0.733. The invalid edges are genuine defects in DMMPP's LifecycleGraphFragment: onResume→onResume (self-loop bug), onStop→onCreateView (skips re-create), onDestroy→onDestroy (onDetach reuses onDestroy sub-sig), onDetach→onAttach.
  • Fragment-hosting app: (13+11)/(13+15) = 0.857.

Ours removes all four via the HSM (ρ bounds the fragment under its host; commit/commitNow splice places bring-up/tear-down at synchronized points) → 24/24 valid = 1.000.

commit vs commitNow (detected, real): 12 apps call commit(), 2 call commitNow() (app.fedilab.openmaps, com.e.ulillgo); the synchronous splice is selected for exactly those 2.

Why FlowDroid is the wrong engine for the “Ours” column

FlowDroid builds its own entry point, so it never sees our reduced DMM; correlating our parameter count against FlowDroid cost gives r ≈ 0 (e.g. the 636-param quicktiles is among the fastest, while the 120-param kick.wpapp runs 427 s). The de-bloat benefit can only show on DMMPP's actual downstream (Androlic) — see the next page.

Precision evidence that does hold

On DroidBench's negative-control apps, FlowDroid emits 2 false positives — BroadcastReceiverLifecycle3 and ServiceLifecycle2 — both leaks reported on infeasible lifecycle orderings, exactly the class of FP the synchronized HSM prunes (RQ2 → precision).

Caveat: app-level recall under our fixed sources/sinks list is not a faithful DroidBench F1 (the list is intentionally small); precision on the negative controls is the defensible signal.

Findings
  • A repeatable regime: 3 of 7 heavy apps (cpustats, watt, smartturnv2) cut time ~50% and peak memory ~17–22% — squarely in the paper's −42%/−45% range. These are parameter-bound analyses.
  • The other heavy apps are neutral (≤3%): their cost is dominated by the number of paths (2k–4k), which de-bloating does not change — an honest boundary of the technique.
  • Path counts are essentially identical baseline vs ours everywhere (watt 890/871, downloader 3791/3739) → de-bloating preserves analysis semantics; the saving is purely smaller per-path symbolic state.

This is the first measured “Ours” RQ3 column on DMMPP's real downstream analyzer. It confirms the direction and, on parameter-bound apps, the magnitude of the paper's claim.

Real de-bloated DMM generator

src/cn/ac/ios/dmmpp/gen/PDMMGen.java · src/cn/ac/ios/pdmm/PDMMGenMain.java · VerifyApk.java

Beyond measuring, PDMMGen emits an instrumented APK whose every component carries a parameter-de-bloated dummy main (the de-bloated analogue of DMMPP's own DMMMain, same dex output). Parameter merging uses the exact-type variant so every emitted argument is type-correct and the dex passes JVM verification.

Cross-check: emitted reduction (66.0%) independently confirms the RQ1 metric (66.2%). On FragmentLifecycle1 the emitted MainActivity.DMMPP signature collapses to DMMPP(android.os.Bundle, boolean[]) — the four lifecycle Bundles merged into one, exactly the paper's Example 2.

Two real DMMPP bugs fixed

The shipped DMMPP.jar throws NullPointerException and produces no output on (a) fragment-bearing apps and (b) apps with non-overridden lifecycle methods — DMMMain has no per-class guard, so one bad component aborts the whole run (verified on FragmentLifecycle1, where the original jar emits nothing). Our generator emits an addressable nop for intermediate join nodes and unresolved lifecycle methods, and guards each component, so it succeeds where DMMPP crashes.

Three concrete recommendations

  1. Lead RQ3 with the parameter-bound cases. cpustats / watt / smartturnv2 give a clean −50% time / −20% memory through the real Androlic engine. Frame them as the “param-bound” regime.
  2. Add a threats-to-validity note: de-bloating is neutral when analysis cost is dominated by path count (thousands of paths), not symbolic-state size — observed on quicktiles / gitsaibot.
  3. To raise the cross-app average toward −42%/−45%, use a fragment-/parameter-heavy app set (or Telegram-scale targets). The current mid-size corpus yields −23%/−7% on average. Note that the bundled FlowDroid 2.10 cannot parse modern (aapt2) large APKs — pin an older app build or upgrade FlowDroid.

What is new versus the original DMMPP

  • Parameter Equivalence Partitioning that actually emits valid, de-bloated APKs (66% fewer entry params).
  • A formal lifecycle spec + ρ map that scores every generator at 1.000 LCS for Ours vs 0.857 for DMMPP on fragments.
  • Two crash-bug fixes that make the generator usable on real fragment apps where DMMPP aborts.
  • The first end-to-end “Ours vs baseline” measurement on DMMPP's real downstream analyzer.
